# Звіт до Лабораторної роботи №3

**Тема:** Налаштування роботи веб-ресурсу для упередження індексації деяких розділів сайту пошуковими сервісами.

## 1. Створення та налаштування сайту
Для виконання роботи було створено базовий сайт з трьох сторінок:
- `index.html` (головна, містить форму з RECAPTCHA)
- `about.html` (про сайт)
- `private.html` (приватна інформація, заборонена до індексації)

Було створено файл `robots.txt` для заборони індексації приватної сторінки:
```txt
User-agent: *
Disallow: /private.html
```

Також було створено файл `.htaccess` для базової автентифікації на рівні веб-сервера Apache:
```apache
<Files "private.html">
    AuthType Basic
    AuthName "Restricted Content"
    AuthUserFile /path/to/.htpasswd
    Require valid-user
</Files>
```
У файл `private.html` також додано мета-тег: `<meta name="robots" content="noindex, nofollow">`.

*(Тут необхідно вставити скріншот коду `robots.txt` та `.htaccess`, а також сторінки сайту)*

## 2. Індексація за допомогою Google Search Console
Сайт необхідно розмістити на безкоштовному хостингу (наприклад, GitHub Pages, 000webhost або InfinityFree), після чого додати його до **Google Search Console**.

*(Тут необхідно вставити скріншоти процесу додавання сайту в Google Search Console та розділу "Покриття", де видно, що URL `/private.html` виключено з індексування)*

## 3. Перелік пентестів на основі Metasploit та Google Dorks

**Мета:** Виявити витоки конфіденційної інформації (паролі, конфігурації, бекапи) з веб-сайту.

### 3.1. Google Dorks для виявлення заборонених/прихованих елементів:
- **Пошук бекапів сайту:**
  `site:example.com ext:sql OR ext:tar OR ext:zip OR ext:bak inurl:backup`
- **Пошук файлів конфігурації (можуть містити логіни/паролі до БД):**
  `site:example.com ext:env OR ext:ini OR ext:conf "DB_PASSWORD"`
- **Пошук директорій, відкритих для індексування (Directory Listing):**
  `site:example.com intitle:"index of" "private"`
- **Пошук файлів журналів (логів):**
  `site:example.com ext:log "error" OR "warning"`

### 3.2. Пентести за допомогою Metasploit:
- **Сканування директорій та файлів:**
  Використання модуля `auxiliary/scanner/http/dir_scanner` для пошуку прихованих або забутих файлів та директорій.
  ```bash
  use auxiliary/scanner/http/dir_scanner
  set RHOSTS example.com
  run
  ```
- **Сканування на наявність відкритих бекапів та копій:**
  Використання модуля `auxiliary/scanner/http/files_dir` для перевірки наявності `.bak`, `.old`, `~` файлів.
- **Аналіз `robots.txt`:**
  Використання модуля `auxiliary/scanner/http/robots_txt` для вилучення вмісту `robots.txt` та автоматичного сканування заборонених у ньому шляхів на предмет доступу.
  ```bash
  use auxiliary/scanner/http/robots_txt
  set RHOSTS example.com
  run
  ```

## 4. Відповіді на контрольні запитання

**1. Роль файлів «.htaccess» і «robots.txt» в пошуку інформацїі сторінки.**
- **`robots.txt`** — це звичайний текстовий файл, який надає пошуковим роботам (краулерам) рекомендації та правила щодо того, які сторінки, файли або директорії дозволено індексувати, а які — заборонено (директиви `Allow` та `Disallow`). Він допомагає уникнути потрапляння службових даних у результати пошуку.
- **`.htaccess`** — це файл конфігурації веб-сервера Apache. На відміну від `robots.txt`, він не просто дає рекомендації роботам, а фізично керує доступом на рівні сервера. За його допомогою можна налаштувати парольний доступ до певних директорій (AuthType), заборонити перегляд вмісту каталогів (Options -Indexes) або налаштувати редиректи, що повністю блокує доступ і для людей, і для пошукових роботів.

**2. Які інструменти пропонує Google для пошуку релевантної інформації?**
Google пропонує такі інструменти:
- **Розширений пошук (Google Dorks)** — використання спеціальних операторів (`inurl:`, `site:`, `intitle:`, `filetype:`, `cache:` тощо) для точкового пошуку прихованої або специфічної інформації.
- **Google Search Console** — інструмент для веб-майстрів, що дозволяє керувати індексацією сайту, перевіряти помилки сканування та аналізувати пошуковий трафік.
- **Google Alerts** — автоматичне сповіщення про появу в мережі нової інформації за заданими ключовими словами (може використовуватись для моніторингу витоків даних компанії).

**3. Які засоби допомагають уникнути витоку даних із ресурсу з відкритим доступом при його відлагодженні.**
- Використання файлу **`robots.txt`** з правилом `Disallow: /` для закриття від пошуковиків усього сайту під час розробки/відлагодження.
- Використання мета-тегів на самих сторінках: `<meta name="robots" content="noindex, nofollow">`.
- Налаштування серверної авторизації через **`.htaccess`** та `.htpasswd` (Basic Auth), щоб сайт могли бачити лише розробники із паролем.
- Використання **CAPTCHA / reCAPTCHA**, що запобігає автоматизованому збору даних ботами.
- Заборона Directory Listing на сервері (щоб зловмисники не бачили список усіх файлів у папках).
- Тестування (staging) на закритому локальному або ізольованому від мережі Інтернет сервері, замість публічного домену.
